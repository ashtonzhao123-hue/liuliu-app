## 一、取消订单 → 温和挽留面板（#7-②）

**优先级**：P0（最高） | **实施顺序**：第1步 | **改动文件**：`src/pages/owner/OrderDetailPage.tsx`

**现状**：取消订单用系统alert"确定取消吗？"，确认即完事，流失无挽留。

**改法**：
- 点击取消 → 不弹系统alert，改为**antd-mobile `Popup`**（bottom sheet，`position='bottom'`）
- 面板内容：
  - 取消原因快选标签（antd-mobile `Selector`，点击即选，不用打字）：
    - `时间不合适` / `找到别人遛了` / `狗狗不舒服` / `临时改主意` / `其他`
  - 选"时间不合适" → 自动弹出挽留条：「要不要换个时间？」+ `Button` 跳转修改时间页
  - 选"找到别人遛了" → 显示「没关系，下次需要随时来」+ 关闭面板
  - 选其他原因 → 正常进入取消流程
- 底部固定挽留条：
  - 文案：「已经有3位遛遛侠看到了你的需求，确定取消吗？」
  - 两个按钮强弱对比：`Button color="primary"`（天青#8DB3C4，文案"再等等"） vs `Button fill="none"`（浅墨灰，文案"还是取消"）
- 取消成功后 Toast：「下次想遛随时来，毛孩子等你呢 🐾」（用 `notify()` 工具函数）

**技术要点**：
- antd-mobile `Popup` 代替 RN 的 bottom sheet
- antd-mobile `Selector` 实现标签选择
- 挽留逻辑用 `useState` 管理选中原因，条件渲染挽留条
- 取消API调用 `cancelOrder()` 在 `src/api/owner.ts:328`，不需要改

**验收**：
- [ ] 点击取消 → 弹出底部面板，不是系统alert
- [ ] 选"时间不合适" → 出现挽留条+"修改时间"按钮
- [ ] 选"找到别人遛了" → 友好关闭，不做挽留
- [ ] 选其他原因 → 正常取消流程
- [ ] 取消成功 → Toast文案是口语化的，不是"订单已取消"

**参考**：滴滴取消挽留；美团外卖取消弹窗

## 二、宠物管理页 → 毛孩子卡片可视化切换（#7-①）

**优先级**：P1（高） | **实施顺序**：第1步 | **改动文件**：`src/pages/owner/PetListPage.tsx`

**现状**：宠物列表/表单形式，切换宠物像切数据不像切家人。

**改法**：
- 每只宠物一张横卡：大头像（圆形）+名字+品种+年龄，**横向 CSS `scroll-snap-type: x mandatory` 滑动切换**
- 选中态：卡片 `transform: scale(1.05)` + `box-shadow` 加深；未选中：`scale(0.9)` + `opacity(0.6)`，用 CSS `transition: transform 0.3s cubic-bezier(0.34, 1.56, 0.64, 1)` 实现 spring 感
- 卡片下方信息区：体重/性格/注意事项用 icon+短文案一行展示
- 底部「+添只毛孩子」按钮：绢白底 + `border: 1px dashed`，暗示"还有位置"
- 空状态：用 `FriendlyEmpty` 组件（`src/components/FriendlyEmpty.tsx`），文案"先认识一下你家毛孩子？"
- 滑动切换动画：卡片间过渡用 CSS `scroll-behavior: smooth`

**技术要点**：
- 横向滑动用 CSS `overflow-x: auto; scroll-snap-type: x mandatory`，每张卡片设 `scroll-snap-align: center`
- 选中态放大/缩小用 CSS `transition`（带 cubic-bezier 模拟 spring），**不用 RN 的 Animated.spring**
- 宠物数据通过 `listPets()` (`src/api/owner.ts`) 获取，`PET_BREEDS` 数组有品种映射
- 品种emoji映射：`比熊→🐕 贵宾/泰迪→🐩 博美→🐕 雪纳瑞→🐕‍🦺 吉娃娃→🐕 约克夏→🐕 柯基→🐕`（暂用通用🐕+文字区分）

**验收**：
- [ ] 多只宠物时横向滑动切换流畅
- [ ] 选中卡片有放大+投影效果
- [ ] 空状态用 FriendlyEmpty 组件
- [ ] 「+添只毛孩子」按钮虚线边框

**参考**：Apple Wallet卡片滑动；Petfinder宠物卡片

## 三、遛狗报告卡 → 订单完成后情感闭环（#4-①）

**优先级**：P1（高） | **实施顺序**：第2步（先DB后前端） | **改动文件**：`WalkerFinishPage.tsx`（遛狗员端）、`OrderDetailPage.tsx`（主人端）

**现状**：遛完只显示"已完成"三个字，主人最想知道的——狗开心吗？拉了吗？遛了多久？一个没回答。

**前置：数据库 Migration**

在 `supabase/` 目录下新建 `add-walk-report.sql`：

```sql
-- Migration: 遛狗报告卡字段
ALTER TABLE orders
ADD COLUMN IF NOT EXISTS report_photos JSONB DEFAULT '[]'::jsonb,        -- 照片URL数组
ADD COLUMN IF NOT EXISTS has_poop BOOLEAN DEFAULT NULL,                   -- 是否大便
ADD COLUMN IF NOT EXISTS has_pee BOOLEAN DEFAULT NULL,                    -- 是否尿尿
ADD COLUMN IF NOT EXISTS walker_note TEXT DEFAULT NULL,                   -- 遛狗员备注
ADD COLUMN IF NOT EXISTS walk_distance FLOAT DEFAULT NULL,              -- 遛狗距离(公里)
ADD COLUMN IF NOT EXISTS walk_duration INT DEFAULT NULL,                -- 遛狗时长(分钟)
ADD COLUMN IF NOT EXISTS report_submitted_at TIMESTAMPTZ DEFAULT NULL;  -- 报告提交时间

COMMENT ON COLUMN orders.report_photos IS '遛狗报告照片URL数组，JSONB格式，最多3张';
COMMENT ON COLUMN orders.report_submitted_at IS '报告提交时间，NULL表示未提交';
```

同时在 `src/api/mappers.ts` 的 `mapOrderRow` 函数中添加对应字段映射，在 `src/types/index.ts` 的 `Order` 接口中添加对应类型。

**改法**：

### 遛狗员端 (`WalkerFinishPage.tsx`)
- 点击"完成遛狗"→**强制弹出报告表单**（antd-mobile `Popup` 全屏），必须填写才能提交：
  - 上传1-3张遛狗照片（antd-mobile `ImageUploader`，必传至少1张）
  - 标记大便/尿尿状态（antd-mobile `Switch` 或两个 `Checkbox`）
  - 一句话备注（antd-mobile `TextArea`，可选，如"柯基今天精力超好🐶"）
- 填写完才能调用 `finishWalkerOrder()` 提交完成
- 提交时把报告数据一并写入 orders 表

### 主人端 (`OrderDetailPage.tsx`)
- 订单完成后（`OrderStatus.Completed`），在页面中展示**遛狗报告卡**：
  - 顶部：照片横滑（CSS `overflow-x: auto; scroll-snap-type: x mandatory`）
  - 中部：路线轨迹缩略图 + 距离 + 时长（"今日遛弯2.3公里，42分钟"）
  - 底部：遛狗员备注 + 💩/尿尿标记（青碧色圆点✓/✗）
- 报告卡视觉：绢白底 + 细线框（`border: 1px solid #E0D8C8`），与订单信息区视觉分层

**验收**：
- [ ] 遛狗员点击完成 → 弹出报告表单（不是直接完成）
- [ ] 至少1张照片必传，否则不能提交
- [ ] 主人端订单完成页能看到报告卡
- [ ] 报告卡有照片横滑、距离时长、便便标记、备注

**参考**：Rover/Wag Post-Walk Report Card

## 五、双盲评价系统（#6-①）

**优先级**：P2（中） | **实施顺序**：第3步（依赖报告卡） | **改动文件**：评价相关页面 + reviews 表

**现状**：评价互相可见，双方怕报复不敢说真话。

**前置：数据库 Migration**

在 `supabase/` 目录下新建 `add-blind-review.sql`：

```sql
-- Migration: 双盲评价系统
ALTER TABLE reviews
ADD COLUMN IF NOT EXISTS is_revealed BOOLEAN DEFAULT FALSE,         -- 双方都评后公开
ADD COLUMN IF NOT EXISTS revealed_at TIMESTAMPTZ DEFAULT NULL,      -- 公开时间
ADD COLUMN IF NOT EXISTS dimension_tags JSONB DEFAULT '[]'::jsonb,  -- 评价维度标签
ADD COLUMN IF NOT EXISTS private_note TEXT DEFAULT NULL;            -- 私信小建议

COMMENT ON COLUMN reviews.is_revealed IS '双方都提交或14天到期后设为true';
COMMENT ON COLUMN reviews.dimension_tags IS '主人评：准时/细心/狗狗喜欢；遛狗员评：温顺/好找/沟通顺畅';
```

**改法**：
- 双方14天内评价，**写完后互相不可见**，都提交（或14天到期）才同时公开（`is_revealed = true`）
- 评价页增加"小建议"私信入口（`private_note`，仅对方可见）
- 评分UI复用项目已有的 `ScorePicker` 组件（`src/components/ScorePicker.tsx`），1-5星评分
- 评价维度拆细：
  - 主人评遛狗员：准时 / 细心 / 狗狗喜欢（antd-mobile `Selector` 多选标签 + 可选文字）
  - 遛狗员评主人：宠物温顺 / 地址好找 / 沟通顺畅
- 评价提交后文案：「你的评价会在对方也写完后一起显示，实话实说就行」
- 显示评价时判断 `is_revealed`，未公开显示"评价将在对方完成后显示"

**验收**：
- [ ] 先评的一方看不到对方的评价内容
- [ ] 双方都评后同时公开
- [ ] 14天到期后单方评价自动公开

**参考**：Airbnb双盲评价机制

## 六、遛狗员战绩面板（#6-②）

**优先级**：P2（中） | **实施顺序**：第4步 | **改动文件**：`WalkerHomePage.tsx`（顶部增加战绩卡片）

**现状**：遛狗员端=接单+订单列表，纯工具感，无成就感。

**改法**：
个人中心/首页顶部增加"遛遛战绩"卡片（`Card className="summary-card"`）：
- 三行核心数据（用 `requestAnimationFrame` 实现数字翻滚递增，不用 RN 的 `Animated.timing`）：
  - 累计遛狗XX次
  - 陪伴XX只毛孩子
  - 遛了XX公里
- 月度徽章体系（icon+名称，灰色=未解锁 / 青碧#5B9A8B=已解锁）：
  - 连续接单7天 → 「风雨无阻」
  - 累计30次 → 「资深遛遛侠」
  - 好评≥4.8 → 「五星侠客」
- 每周自动生成「遛遛周报」卡片：
  - 本周遛了X只狗 + X公里 + 收到X条好评
  - 底部小字"遛遛侠·让每只狗都开心"
  - 分享用 `navigator.share()` Web Share API（分享文本+链接），**不支持图片分享**

**技术要点**：

## 七、天气感知时间选择器（#8-①）

**优先级**：P2（中） | **实施顺序**：第4步 | **改动文件**：`src/pages/owner/CreateOrderPage.tsx`

**前置依赖**：需先在 `.env` 中配置 `VITE_HEFENG_WEATHER_KEY=你的和风天气API Key`（免费版：https://dev.qweather.com/，注册即得）

**现状**：主人选遛狗时间全凭感觉，夏天下午2点地面温度50-60℃。

**改法**：
- 时间选择器上方增加天气感知条，一行三列：当前气温 | 体感温度 | 地面温度预估
- 天气数据通过和风天气 API 获取（前端 fetch）：`https://devapi.qweather.com/v7/weather/now?location={经度,纬度}&key={API_KEY}`
- 根据天气自动标注时段安全性：
  - 🟢 青碧#5B9A8B「适合遛弯」——温度适宜(15-28℃)
  - 🟡 缃色#E8C547「有点热，建议傍晚再来」——地面温度>40℃
  - 🔴 浅赭石#B5836A「太热了，这个时间别遛」——地面温度>50℃，该时段 `disabled`
- 冬天反向：早晨<5℃标注「太冷了，等太阳出来」
- 文案口语化：「现在外面28℃，挺舒服的」而非「当前温度28℃」
- 选中危险时段时底部提示条 + `Button`「调整时间」

**技术要点**：
- 地面温度预估公式：`气温 × 1.5 ~ 2.0`（经验值）
- 时段安全等级判定：安全(15-28℃) / 注意(地面>40℃) / 危险(地面>50℃禁选)
- 用 antd-mobile `DatePicker` 或自定义时间选择器
- 天气API请求失败 → 静默降级，不显示天气条（不阻塞用户）

**验收**：
- [ ] 天气条正确显示气温/体感/地面温度
- [ ] 高温时段标注为不可选或警告
- [ ] API 请求失败不阻塞正常使用
- [ ] 文案口语化

**参考**：美团外卖恶劣天气提醒

## 八、骨架屏全局加载态（#8-②）

**优先级**：P0（最高，基建） | **实施顺序**：第0步（最先做） | **改动范围**：全局所有列表/详情页

**现状**：PWA弱网场景频繁，白屏等2秒用户直接退出。

**改法**：

### Step 1：封装通用 Skeleton 组件（新建 `src/components/Skeleton.tsx`）

```typescript
// SkeletonCard / SkeletonAvatar / SkeletonText / SkeletonImage
// 每个组件接受 width/height/className props
// 动画用 CSS @keyframes shimmer：绢白#F4ECD8 → 浅墨灰 → 绢白 横向渐变流动
// 最小展示300ms（防闪烁），内部用 useEffect + setTimeout
```

CSS 动画：
```css
@keyframes shimmer {
  0% { background-position: -200% 0; }
  100% { background-position: 200% 0; }
}
.skeleton {
  background: linear-gradient(90deg, #F4ECD8 25%, #E8E0D0 50%, #F4ECD8 75%);
  background-size: 200% 100%;
  animation: shimmer 1.5s infinite;
}
```

### Step 2：各页面骨架屏

1. **遛狗员列表页**（`WalkerHomePage.tsx`）：SkeletonCard × 3 + SkeletonAvatar 圆形 + SkeletonText 横条
2. **订单详情页**（`OrderDetailPage.tsx`）：模拟报告卡布局占位
3. **个人中心**：头像圆形占位 + 数据三列占位

### Step 3：兜底处理
- 5秒未加载 → 显示错误提示 + 重试按钮（antd-mobile `ErrorBlock`）
- 离线时：骨架屏 + 顶部提示条「当前离线，可查看已缓存内容」

**技术要点**：
- **不用 spinner 转圈**，用 shimmer 骨架屏
- 骨架屏结构必须和真实布局一致（位置/比例/层级），改布局时同步更新骨架屏
- 数据到达后用 CSS `opacity` transition 淡入替换（300ms）
- 列表页可复用 `useScrollReveal` hook（`src/hooks/useScrollReveal.ts`）：数据加载完成后，给列表项加 `data-reveal` 属性实现逐个淡入（stagger动画）

**验收**：
- [ ] 各页面加载时显示骨架屏而非白屏
- [ ] shimmer动画流畅
- [ ] 数据到达后骨架屏淡出、内容淡入
- [ ] 5秒超时后显示重试按钮

**参考**：Airbnb首页全骨架屏；骨架屏比spinner降低30%感知等待时间（LogRocket研究）

## 全局约束

1. **配色**：严格遵循宋式美学（见上文配色表），禁止高饱和色/纯白纯黑
2. **文案**：全部口语化去AI味，Toast用 `notify()` (`src/utils/notify.ts`)，错误用 `getFriendlyErrorMessage()` (`src/utils/errors.ts`)
3. **动画**：CSS `transition` + `@keyframes`（优先 cubic-bezier 模拟 spring），交互反馈 300ms 内，数据更新用淡入/翻滚动效
4. **组件**：优先使用 antd-mobile 已有组件（`Popup`/`Card`/`Button`/`Selector`/`ImageUploader`/`TextArea`/`Switch`/`ErrorBlock`等），不要引入新的UI库
5. **数据库变更**：需要新增字段的先写 migration SQL 到 `supabase/` 目录，再改 `mappers.ts` 和 `types/index.ts`
6. **类型安全**：所有改动必须在 `src/types/index.ts` 中同步更新接口定义
7. **不引入新依赖**：除非标注"前置依赖"（如天气API Key），否则只用已有依赖
8. **路由**：新页面记得在 `App.tsx` 中注册路由
